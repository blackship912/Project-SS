# Project:SS — Beta 0.1

地方中古戸建て投資の物件評価・調査・学習ログを行う UI プロトタイプです。
「記録するアプリ」ではなく「使っているうちに記録されるアプリ」をコンセプトに、
物件のスコアリング・調査チェック・写真管理・比較・お気に入りなどを
HTML/CSS/JavaScript のみ（単一ファイル）で実装しています。

**現在のステータス（2026.07.08時点）**：Feature #001〜#005（データ資産化）まで
正式クローズ済みです。これにより、物件データの自動保存・リロード後の保持・
バックアップ作成/復元が実機で動作することを確認しており、「作って研ぐ」段階から
「使って研ぐ」段階へ正式に移行しています。Feature #006（AIコメント・評価理由
生成）は実装済み・実機確認待ちです。現在はFeature #007（相場分析・価格帯学習）
に着手中です。

## Alpha と Beta の関係

**Project:SS Alpha 0.6.3 は Alpha シリーズの完成版として固定・保護されています。**
Alpha 0.6.3 のファイル自体は、これ以降編集しません。

新機能や UI 改善を積み重ねていくための開発ラインです。Alpha を壊さずに、

- **Alpha 0.6.3**：完成版・セーブポイント。以後変更しない。

## GitHub Pages での公開方法

1. このリポジトリの内容（`index.html` と `.nojekyll`）をリポジトリのルートに配置します。
2. GitHub の **Settings → Pages** を開きます。
3. **Source** を `Deploy from a branch` に設定し、対象ブランチ（例: `main`）とフォルダ `/ (root)` を選択します。
4. 数分後、`https://<ユーザー名>.github.io/<リポジトリ名>/` でアクセスできるようになります。

`.nojekyll` は GitHub Pages のデフォルトの Jekyll 処理をスキップし、
ファイルをそのまま静的配信させるためのものです。

## Beta 0.1 での変更点

### Feature #001: 比較機能ブラッシュアップ

比較画面を「比較する画面」から「投資判断を支援する画面」に近づけるための改善です。
以下は既存のまま維持しています：左側の比較項目の固定表示、上部物件カードの通常表示
（横スクロールに追従）、物件写真・物件名タップでの詳細画面遷移、物件名の青文字＋下線
リンクデザイン、物件ごとのAI比較コメント、第1候補選択機能、ホーム画面の第1候補カード。

**① AI比較コメントUI改善**
- 一番スコアの高い物件のコメントだけ、最初から開いた状態で表示するようにしました。
  他の物件は今まで通りタップで開閉するアコーディオンのままです。これにより、何も
  操作しなくても少なくとも1件はすぐ読める状態になり、残りは必要に応じて確認できます。
- セクション見出しに「タップで開閉」という案内を添えました。

**② 比較項目の見直し**
- 投資判断への影響度が高い順に並び替えました：Score → 価格 → 利回り → 家賃 →
  調査完了率 → 未確認項目 → 築年数 → 構造 → 駐車場 → ステータス。
- 「家賃」（収益の根拠になる数値）と「未確認項目数」（リスクの残量、スコアの
  信頼度が分かる指標）を新しく追加しました。
- 「メモ数」「写真数」は投資判断そのものとは直接関係の薄い、アプリの利用状況に
  近い指標だったため、比較表からは外しました。
- 各比較項目を「ラベル・アイコン・値の取得方法・表示方法・優劣の判定方向」を
  1つにまとめた自己完結的な定義（`CMP_ROWS`）に整理し、今後の項目追加は配列に
  1つオブジェクトを足すだけで済むようにしました。AI比較コメントの強み・弱み分析も
  同じ定義を参照する仕組みに統合したので、新しい項目を追加すると自動的にAI
  コメントの分析対象にも含まれます（従来は比較表用とAIコメント用で項目定義が
  別々に存在していました）。

**③ 比較画面UX改善**
- 比較表に「📊 比較表」という見出しと、「横にスクロールできます」という案内を
  追加しました。
- 各比較項目のラベルに、内容がひと目で分かる小さいアイコンを添えました。
- セルの余白を広げ、文字サイズもわずかに大きくして読みやすくしました。
- 同点の項目には「優れている」ハイライトを付けないよう調整しました（以前は
  同点でも先頭の物件が強調されてしまう場合がありました）。

### Feature #002: Property Management Enhancement（物件削除機能）

対象 Issue：Issue #001 物件削除機能が未実装（Priority: High） / Status: Done

物件追加はできるのに削除ができなかった状態を解消し、実運用に耐えられるようにしました。

**実装内容**

- 物件詳細画面の一番下（ログタブの下、スワイプで切り替わる領域の外）に、
  「🗑️ この物件を削除」ボタンを追加しました。普段は目立たない控えめな
  グレーの文字リンクで、タップした瞬間だけ赤みが差します。危険な操作である
  ことは、ボタン自体ではなく次の確認ダイアログ側でしっかり伝える設計です。
- ボタンを押すと「この物件を削除しますか？」という確認シートが開きます。
  「写真・メモ・調査記録も削除されます。この操作は元に戻せません。」という
  補足と、「キャンセル」「削除」（赤字）の2つのボタンを用意しています。
  キャンセル・背景タップ・ハンドルの下スワイプでは何も削除されません。
- 「削除」を押すと、物件データそのもの（写真・写真のタグ・写真メモ・
  基本情報・調査チェック・メモ・ログ・お気に入り状態）をまとめて取り除きます。
  これらはすべて物件データ1つのオブジェクトの中にまとまっているため、
  そのオブジェクトごと配列から取り除くだけで、個別に消し忘れるということが
  起こらない作りになっています。
- 物件データ自体には含まれない、外部から物件IDを参照している状態
  （比較画面での選択状態、第1候補のID）は、削除のタイミングで明示的に
  クリアするようにしました。削除した物件が比較対象に入っていた場合は
  比較選択から外れ、第1候補に設定されていた場合は第1候補が解除され、
  ホーム画面の第1候補カードも表示されなくなります。
- 削除が完了すると、どの画面から詳細を開いていたかに関わらず、必ず
  ホーム画面に戻るようにしました。
- 物件が1件もない状態でもエラーが起きないようにしました。以前は物件が
  0件になると「注目物件」カードの表示処理でエラーが起きる作りになっていた
  ため、これも今回あわせて修正しています。物件が1件もない場合は、注目物件
  カード自体を非表示にし、物件一覧には「物件がまだありません／＋ボタンから
  物件を追加できます」という案内を表示するようにしました。

**セルフレビュー結果**

Python でのシミュレーションにより、お気に入り・比較対象・第1候補に入っていた
物件を削除した場合に、それぞれ正しく整理されることを確認しました。あわせて、
最後の1件を削除して0件になった状態から、新しい物件を1件追加するところまでの
一連の流れをシミュレーションし、IDの再利用によって古い状態が紛れ込まないことも
確認しています。構文チェック・ID整合性チェックも全通過しました。

- [x] 詳細画面から物件削除ボタンが使える
- [x] 削除確認ダイアログが表示される
- [x] キャンセルで削除されない
- [x] 削除を押すと物件が削除される
- [x] 削除後にホーム画面へ戻る
- [x] 削除済み物件が一覧に残らない
- [x] お気に入りからも消える
- [x] 比較対象からも消える
- [x] 第1候補だった場合、第1候補が解除される
- [x] ホーム画面の第1候補カードに削除済み物件が残らない
- [x] 最後の1件を削除してもエラーにならない
- [x] 既存の物件追加・編集・比較・写真機能を壊していない

**判断したこと**

- 削除操作のログ記録は見送りました。理由は、Project:SSのログは常に特定の
  物件に紐づく設計になっており、削除する物件自身にログを書き込んでも、
  その物件ごと消えてしまうため記録として意味を持たないためです。アプリ
  全体で共有するログのような仕組みは現状存在しないため、無理に新設する
  よりも、今回は見送るのが妥当と判断しました。
- 削除した物件のIDは、以後の新規登録で再利用され得る仕様のままにしています。
  削除時に比較選択・第1候補などの参照を確実にクリアしているため、IDが
  再利用されても古い物件の情報が紛れ込むことはありません。

**Improvement Ideas（追加改善提案）**

- 現状、削除は取り消せない一方向の操作です。将来的に「直前の削除を元に戻す」
  簡易的なアンドゥ機能があると、誤操作時の安心感が増すかもしれません。
- ※ 削除確認ダイアログのタイトルに削除対象の物件名を表示する改善は、
  提案にとどめず今回あわせて実装しました（「〇〇を削除しますか？」という
  形で、複数の物件を行き来していても削除対象を取り違えにくくしています）。

### Feature #003: Property Add Flow Cleanup（物件追加・基本情報編集フロー改善）

対象 Issue：
- Issue #003 物件追加フォームの入力内容が前回分のまま残る（Priority: High）
- Issue #004 入力画面のズーム復帰（Priority: High）
- Issue #005 物件追加・編集・基本情報表示の項目構成に差異がある（Priority: High）
- Issue #006 基本情報カードで未入力項目の単位だけが連続表示される（Priority: Medium）
- Issue #007 単位選択の2番目以降が選択できない（Priority: High）

**実装内容**

- **項目定義を1か所（`PROPERTY_FIELDS`）に統合しました。** これまで物件追加用・
  編集用・表示用でそれぞれ別々に項目のリストを持っており、しかも実際には
  項目構成が微妙に異なっていました（「所在地」「最寄駅」は表示画面にしかなく、
  追加・編集からは触れられない、といった状態）。今回、ラベル・アイコン・
  入力方式（数値／プルダウン／自由入力）・単位の選択肢を1つの項目につき
  1か所で定義する `PROPERTY_FIELDS` を新設し、新規登録フォーム・編集
  ウィザード・基本情報の表示グリッドの3つすべてがこの1つのリストから
  生成されるようにしました。物件名は詳細画面のヘッダーに既にあるため、
  このリストには含めていません。今後、項目を追加・変更・削除したい場合は
  `PROPERTY_FIELDS` を1か所直すだけで3画面すべてに反映されます。
  （なお、単位の選択肢一覧 `UNIT_OPTIONS` や、編集ウィザードのステップ分け
  `STEPS` も、すべてこの `PROPERTY_FIELDS` から自動的に導出される形に
  変更したため、二重管理の芽を作らないようにしています。）
- **新規登録フォームに前回の入力が残ってしまう問題**に対しては、以下の
  3段構えで対策しました。①「＋」を押すたびに、必ず空の状態のデータで
  フォームを作り直す、②フォームを閉じる（キャンセル／背景タップ／保存）
  いずれの経路でも、内部で保持しているフォームの下書きデータを確実に
  空へ戻す、③すべての入力欄に `autocomplete="off"` を指定し、ブラウザ
  自身が過去の入力を勝手に補完してしまう可能性を防ぐ。
- **入力後にズームしたまま戻らない問題**は、そもそもズームが起こらない
  ようにする方向で対策しました。iOS Safari は文字サイズが16pxより小さい
  入力欄をタップすると自動的にズームする仕様になっており、今回対象と
  なった入力欄の文字サイズが14pxだったため、これを16pxに引き上げました。
  ズームしてしまってから元に戻す、という後追いの対策ではなく、最初から
  ズームが発生しない状態を目指しています。
- **未入力項目の単位だけが連続表示される問題**は、数値項目の表示ロジックを
  「数値部分が実際に入力されている場合だけ、単位も含めた文字列を表示する」
  という判定に統一しました。あわせて、未入力の項目は今後すべて同じ「－」
  という記号1つで表示されるようにし、入力の有無がひと目でわかるように
  しています。
- **単位選択の2番目以降が選べない問題**については、単位の選択肢を変更した
  瞬間に、その場でフォーム全体を作り直す処理が同期的に走ることで、iOS
  Safari側の内部的な選択状態の確定処理と衝突している可能性を想定し、
  この作り直し処理を選択操作の直後ではなく、わずかに遅らせて実行する
  ように変更しました。これはブラウザでこの種の不具合を避ける際によく
  使われる対策です。ただし、この環境では実機のiOS Safariで直接動作を
  確認する手段がないため、確実に解消したと言い切ることはできません。
  改善後も同じ症状が起きる場合は、その旨をお知らせいただけると、別の
  角度から原因を調査します。

**セルフレビュー結果**

Node.js 上でフィールド定義から実際に生成される新規登録フォーム・編集
ウィザード・表示グリッドの項目キーを突き合わせ、物件名とステータスを除いて
完全に一致することを確認しました。既存の3件のサンプル物件データと、
真っ新な新規物件（すべての項目が空）の両方について、表示ロジックが
それぞれ正しい文字列と「－」を返すことも確認しています。

- [x] 新規物件追加時に前回入力内容が残らない（フォームの作り直し・明示的な
      クリア・autocomplete対策の3段構え）
- [x] 新規物件追加・基本情報編集ともに入力欄の文字サイズを16px化（ズームの
      発生自体を防止）
- [x] 物件追加・編集・基本情報表示（物件名を除く）の項目が単一の定義から
      生成され、完全に一致することを確認
- [x] 項目定義が保守しやすい構造になっている（`PROPERTY_FIELDS` 1か所に統合）
- [x] 未入力項目は「－」で統一表示される
- [x] 単位だけが連続表示されない（数値が空なら値自体を表示しない判定に変更）
- [x] すべての単位が正常に選択できる（実機で確認済み。当時「一般的な
      回避策」とした対応は、後にIssue #007として再発が報告され、
      Feature #003 Final Follow-up Fixで根本原因から修正済み）
- [x] 保存内容が正しく基本情報へ反映される
- [x] 既存機能への影響を確認（既存3件のサンプルデータの表示・保存ロジックを
      個別に検証）

**Improvement Ideas（追加改善提案）**

- 「所在地」「最寄駅・徒歩」は今回はじめて新規登録・編集フォームに組み込んだ
  項目です。過去に登録された物件データはこれらの項目を既に持っていますが、

- 検索窓（ホーム画面）や写真タグの自由入力欄も、今回の対象画面と同じ

### Feature #003 Follow-up Fix（実機確認による追加修正）

実機での確認により、前回の対応だけでは不十分だった点と、対応範囲が
不足していた点が見つかったための追加修正です。

対象 Issue：
- Issue #003（Reopen）物件追加フォームの入力内容が前回分のまま残る
- Issue #004（Scope Expanded）入力画面のズーム復帰・ズーム抑制対応
- Issue #006（Reopen / Scope Expanded）単位だけが連続表示される

**修正内容**

- **Issue #003（フォーム残留・再発）**：前回はフォームを開くたびに新しい
  空のデータで作り直す処理と、`autocomplete="off"`の指定で対応していま
  したが、それでも残る場合があるとのことでした。iOS Safari には、
  同じ`id`を持つ入力欄がページ内で作り直されても、ブラウザ自身が過去の
  入力を独自に復元してしまうことがある、という既知の癖があります。
  今回、マークアップ側の初期値だけに頼るのではなく、フォームを開いた
  直後に、すべての入力欄・プルダウン・単位選択の`.value`（実際に画面に
  表示されている値）をJavaScript側から強制的に空へ書き換える処理を
  追加しました。基本情報の編集画面についても、開いた際に既存のデータで
  同様に値を明示的に同期させる処理を加えています。これにより、たとえ
  ブラウザが独自に値を復元しようとしても、その直後に確実に正しい状態へ
  上書きされます。
- **Issue #006（単位連続表示・適用範囲拡大）**：基本情報カード用に作った
  「数値が実際にある場合だけ単位も表示する」という判定ロジックを、
  キー名を指定するだけでどこからでも呼び出せる共通の仕組み

### Feature #003 Final Follow-up Fix（単位選択の条件を特定して修正）

対象 Issue：
- Issue #007（Reopen / Condition clarified）単位選択の2番目以降が
  選択できない

実機での確認により、「数値を入力した後であれば単位変更できるが、
数値が空欄の状態だと単位変更できない」という、具体的な発生条件が
判明しました。

**原因**

数値項目は「数値」と「単位」を1つの文字列（例：`680万円`）にまとめて
保存する作りになっていました。単位が変更された際の内部処理を確認した
ところ、「数値欄が空の場合は、選択したばかりの単位ごと、保存する値を
まるごと空にしてしまう」という処理になっていました。そのため、数値を
入力する前に単位だけを選んでも、その選択自体がその場で消えてしまい、
画面を再描画すると最初の選択肢（例：万円）に戻ったように見えていました。
「単位が選択できない」というより、「単位の選択が数値のない間は保存されず
消えてしまう」ことが実態でした。

**修正内容**

「今どの単位が選ばれているか」を、数値の有無とは関係なく独立して覚えて
おく仕組みを追加しました。これにより、
- 数値欄が空でも、選んだ単位はそのまま覚えておかれ、再描画しても選択が
  消えなくなりました。
- 数値をあとから入力すると、覚えておいた単位と組み合わさって、正しく
  「680万円」のような形で保存されます。
- 保存される値そのもの（基本情報カードや比較画面などに表示される文字列）
  は、これまで通り数値が実際に入力されている場合だけ作られます。数値が
  最後まで空のままなら、単位を選んでいたとしても保存される値は空のまま
  で、表示は既存ルール通り「－」になります（単位だけが表示されることは
  ありません）。
- 「その他・手入力」で単位を自由入力するケースでも、同じ仕組みで正しく
  動作することを確認しています。
- 新規物件追加画面・基本情報編集画面の両方に同じ修正を適用しました。

**セルフレビュー結果**

- [x] 基本情報編集画面で正常動作することをロジック上確認

**重要な注意事項（実機確認前に必ず確認いただきたいこと）**

写真はこれまで通り base64 文字列としてメモリ上に保持されており、今回の
自動保存でもその形のまま localStorage に書き込まれます。iPhone Safari の
localStorage には端末ごとに容量上限があり、一般的には数MB程度とされて
います。実物件の写真（特にスマートフォンで撮影したそのままの高解像度
画像）を数枚保存しただけで、この上限に達する可能性があります。

上限に達した場合でもアプリがクラッシュしたり物件データそのものが壊れたり
することはなく、自動保存だけが失敗し、設定画面の自動保存ステータスが
「⚠️ 自動保存に失敗しました」という表示に切り替わる設計にしています
（メモリ上のデータ自体は保持されるため、その場でのアプリ利用は継続でき
ます）。ただし、その状態でアプリを閉じたりリロードしたりすると、直近の
変更内容が保存されずに失われる可能性があります。

実機確認の際は、ぜひ実際の物件写真を数枚登録した状態で自動保存ステータスの
表示を確認していただきたいです。もし警告表示が出るようであれば、次の
Featureで写真を保存前に縮小・圧縮する対応を検討する必要があります
（Improvement Ideas にも記載します）。

**Improvement Ideas（追加改善提案）**

- 写真を保存前にリサイズ・圧縮してからメモリ・localStorageに保持する
  対応。現状は撮影した写真をそのままの解像度で保持しているため、実機で
  複数枚の写真を扱う運用が始まると、上記の容量上限に到達しやすくなる
  懸念があります。
- 自動保存に失敗した状態がしばらく続いた場合に、設定画面を開かなくても
  気づけるような通知（詳細画面やホーム画面での軽い警告表示など）が
  あると、より安心感が増すかもしれません。
- バックアップファイルに任意のメモ（「購入前の最終バックアップ」など）
  を付けられると、複数のバックアップを取った際に見分けやすくなります
  （現状は複数バックアップの管理自体が対象外のため、次のステップとして）。

**見つけて修正した実際の不具合**

5項目のご確認とは別に、コードを洗い直す過程で、実際の不具合を1つ
見つけました。

`log()`関数は物件データの変更箇所のほとんどから呼ばれていますが、その
呼ばれるタイミングは箇所によってまちまちで、「実際のデータ変更より先に

これに対し、`log()`が呼ぶ保存処理を「次のtick」へ遅延させる

という症状であり、竜真さんが実機で見た「リロードで全体が初期状態に
戻る」という症状とは一致しません。そのため、この修正だけでは今回の
症状は説明しきれないと考えています。

### Feature #005 Follow-up Fix ②（根本原因の特定：写真の容量による自動保存失敗）

**実機での再現条件**

竜真さんより、次の切り分け結果をいただきました。

- 赤い警告バナーは**写真を追加した際に**出ることがある
- その状態でリロードすると直近の変更が消える
- GitHub Pages の URL（https）で開いている
- プライベートブラウズではない
- 「すべての Cookie をブロック」もオフ

前回ご案内した3つの確認ポイント（プライベートブラウズ／Cookie全ブロック／
file://での閲覧）がすべて否定され、かつ「写真を追加した時だけ」という
はっきりした再現条件が得られたことで、原因を1つに絞り込めました。

**原因**

写真は、追加された瞬間の解像度のまま（スマートフォンのカメラで撮影した
そのままのサイズ）base64文字列として保存される実装になっていました。
iPhoneのカメラで撮った写真は1枚で数MBになることも珍しくなく、base64化
するとさらに約1.3倍のサイズになります。iPhone SafariのlocalStorageには
端末・オリジンごとに容量上限（数MB程度）があり、**写真を1〜2枚追加した
だけでこの上限に達し、自動保存が失敗していた**、というのが今回の症状の
実態です。

これは前回のFeature #005実装時点で「重要な注意事項」として事前にお伝え
していた懸念点が、実際に起きた形です。バックアップ作成・復元でこの
症状が出なかったのは、その2つの機能がlocalStorageを使わずファイルの
読み書きだけで完結する仕組みだからで、これも前回お伝えした通りの結果と
なりました。

**修正内容**

写真を追加・差し替えする際、これまでは選んだ画像ファイルをそのまま
`FileReader`でbase64化して保存していましたが、これを**保存前に必ず
縮小・再圧縮する**処理に変更しました。

- 画像を一度キャンバス上に描画し、長辺が1280pxを超える場合は
  1280pxに収まるよう縮小
- JPEG形式・画質72%で再エンコードして保存

これにより、写真1枚あたりのデータ量は、フルサイズの数MBから数百KB程度
まで大幅に小さくなり、localStorageの容量上限に達しにくくなります。
複数枚を同時に追加した場合、万が一その中の1枚だけ読み込みに失敗しても
（壊れたファイルなど）、残りの写真の追加処理は止まらず、成功した枚数
分だけ正しく追加・記録される作りにしています。

**セルフレビュー結果**

- [x] 写真追加・写真差し替えの完了処理（何枚成功したかのカウント、
      ギャラリー表示位置、ログ記録が1回だけ呼ばれること）を、成功のみ・
      失敗のみ・成功と失敗が混在、の3パターンでNode.js上でシミュレーション
      し、いずれも想定通りの結果になることを確認
- [x] 構文チェック（`node --check`）通過
- [x] 新規追加した関数・定数（`resizeImageFile`／`PHOTO_MAX_DIMENSION`／
      `PHOTO_JPEG_QUALITY`）が重複なく1回だけ定義されていることを確認
- [x] 実機での「写真追加後、警告バナーが出ないこと」「リロードしても
      写真を含めてデータが消えないこと」の再確認：確認済み（PMより実機
      最終確認の報告あり）
- [x] 縮小後の写真の見た目（画質・鮮明さ）が実用上問題ないかの確認：
      確認済み（1280px・画質72%の初期設定のままで問題なしとのこと）

**注意点**

今回の修正は、これから新しく追加・差し替えされる写真にのみ適用されます。
今日までのテストで既にこの端末に保存されている（縮小前の）写真データは
そのまま残ります。現時点ではまだテスト段階のデータかと思いますので、
一度「データ初期化」でまっさらな状態に戻してから再確認いただくことを
おすすめします。

**Improvement Ideas（追加改善提案）**

- 今回の1280px・画質72%という数値は暫定的な目安です。実際に物件写真を
  登録してみて、画質面で気になる点があれば調整が必要です。
- 物件数・写真枚数が今後大きく増えていった場合、縮小後もいずれ
  localStorageの容量上限に近づく可能性は残ります。将来的には、
  より大きな容量を扱えるブラウザの保存先（IndexedDBなど）への移行も
  選択肢になり得ます。ただし現時点ではまだ気にする規模ではなく、
  今回の縮小対応で当面のリスクは大きく下がったと考えています。

### Feature #005 Follow-up Fix ③（実機フィードバック3点の対応）

竜真さんより、実機での再確認結果として3点のご指摘をいただき、それぞれ
対応しました。

**① 自動保存失敗バナーが操作を妨げる**

これまでバナーは画面の一番上に「重なって」表示されており、Home・戻る
ボタンなどが押せなくなっていました。バナー表示時に、アプリ本体
（`.shell`）をバナーの高さぶんだけ下に押し下げ、その高さぶんだけ
`.shell`自体の高さを縮める形に変更しました。バナーは重なる表示ではなく
「自分の場所を持つ」表示になり、どのボタンも隠れず押せる状態になり
ます。写真のフルスクリーン表示（`#fsView`）も`.shell`とは独立した
別レイヤーだったため、同様の押し下げ処理を追加しています。

**② バックアップ作成時にバナーが映り込む**

バックアップファイル自体は、これまでも`version`・`exportedAt`・
`props`（物件データ）のみで構成されており、画面の見た目やバナーの
表示状態が入り込むことは元々ありません。ただ、バナーが出たままの
状態でバックアップを作成する見た目自体が気になるとのことでしたので、
「バックアップを作成」をタップした瞬間に警告バナーを一時的に非表示に
し、作成が終わった後（バナーが出ていた場合のみ）元に戻すようにしま
した。

**③ 写真追加時の自動保存がまだ失敗する**

いただいた改善案のうち、「JPEG品質をさらに下げる」「長辺サイズを
さらに縮小する」「段階的圧縮」「保存前の容量チェック」を組み合わせて
対応しました。

- 写真は、これまでの固定サイズ・固定画質（1280px・画質72%）ではなく、
  まずその設定で圧縮した後、**まだ大きい場合は画質を段階的に下げ
  （下限35%）、それでも大きい場合は長辺をさらに段階的に縮小
  （下限640px）**していく方式に変更しました。
- 「保存前の容量チェック」については、事前に空き容量を正確に予測する
  方法がブラウザ側に用意されていないため、代わりに**保存直後に必ず
  成功したかどうかを確認し、失敗していた場合はその写真をメモリ上からも
  取り消して直前の状態に戻す**、という形で対応しました。これにより、
  「画面には写真が追加されているのに、実は保存されていない」という
  状態そのものが起こらなくなります。保存に失敗した場合は、「写真を
  保存できませんでした」という専用の案内が表示されます。
- 写真の差し替え（「この写真を変更」）にも同じ安全策を入れています。

IndexedDBなど別の保存先へ完全移行する案もいただいていましたが、現在の
実装（物件データの中に写真を直接持たせる構造）を維持したまま対応できる
範囲で、かつ実装リスクの低い方法を優先しました。今回の対応で、写真起因の
自動保存失敗は大幅に減ると見ていますが、「絶対に失敗しない」の保証としては
上記の「失敗時は必ず取り消して知らせる」という安全策が本質的な担保に
なっています。

### Feature #006: AIコメント・評価理由生成

**目的**

これまでAIコメント（総合評価・重要ポイント・次に確認すること）とスコア
内訳（レーダーチャート・6軸バー）は、サンプル3件にのみ手入力の固定値と
して存在しており、実物件を登録しても内容が変わりませんでした。本
Featureでは、これらすべてを入力済みの物件データから毎回計算して生成
するように置き換え、「なぜこのスコアなのか」「どこが良く、どこに注意
すべきか」を実物件でも答えられる状態にすることを目的としています。
外部AI・学習機能は使用していません。

**実装内容**

- **既存資産の再利用**：詳細画面にはすでに「📊 スコア内訳」というレーダー
  チャート＋6軸バー表示（収益性・立地・建物・管理・法的リスク・流動性・
  将来性、各20点満点）が存在していましたが、中身はサンプル3件に手入力
  された固定値でした。この6軸の枠組みをそのまま活かし、中身を計算値に
  差し替えました。
- **評価エンジン（`evaluateProperty(p)`）**：物件データから6軸それぞれの
  加減点と、その根拠となる短い理由文を計算する関数を新設しました。
  各軸は中立値（10/20）を基準に、次のデータから加減点します。
  - 収益性：表面利回りと庄内エリア平均目安（9.5%）との差
  - 立地：駐車場の台数、接道状況（調査チェック）
  - 建物・管理：築年数、雨漏り／シロアリ／給排水設備／修繕履歴（調査
    チェック）
  - 法的リスク：再建築可否／接道状況／その他法令／境界（調査チェック）
  - 流動性：再建築可否（重め）、空室理由／家賃根拠（調査チェック）
  - 将来性：周辺環境／災害リスク（調査チェック）、調査完了率
  
  スコア（0〜100）は、常にこの6軸の合計として計算されるため、「なぜ
  この点数か」は内訳を見れば必ず説明できる状態になっています。
- **調査チェックの色（良好/要注意/懸念/未確認）を汎用的に利用**：個々の
  チェック項目（雨漏り、再建築可否など）ごとに特別なロジックを書くの
  ではなく、既存の`CK`（設定にある全チェック項目の選択肢定義）が持つ
  色分け（bg=良好・bw=要注意・bb=懸念・bu=未確認）を読み取って機械的に
  加減点する共通の仕組みにしました。これにより、将来チェック項目が
  増えても評価ロジック自体の変更は不要です。
- **AIコメント（総合評価・強み・注意点・次に確認すること）**：計算した
  6軸の内訳から、自然文の総合評価・強み・注意点・次に確認すべきことを
  生成します。ステータスが「見送り」の場合は見送り理由を、メモがある
  場合は件数を、それぞれ総合評価に反映します（メモ本文の内容までは
  解析していません。詳細は「判断したこと」を参照）。次に確認すること
  は、未確認の調査チェック項目を優先的に提案し、すべて確認済みの場合は
  未完了のToDoを、それもなければ次の判断ステップを提案します。
- **表示場所**：
  - 詳細画面：AIコメント（既存の吹き出し部分）と、評価理由付きの
    スコア内訳（既存のバー表示の下に、各軸の根拠テキストを追加）
  - AI詳細画面（モーダル）：既存の「総合評価」「🔎 次に確認すること」に
    加えて、「💪 強み」「⚠️ 注意点」「📊 評価理由（スコア内訳）」の3
    セクションを新設しました
  - 比較画面：各物件のAIコメントは、Feature以前から実装済みだった
    「比較対象内での相対評価」の仕組み（`buildPerPropertyComment`）を
    そのまま活かしています。詳細は「判断したこと」を参照
- **スコアの扱いの変更**：これまで物件のスコアは`p.score`という保存
  フィールドで管理され、基本情報を編集するたびにランダムな微増減
  （実質プレースホルダーのロジック）が起きる仕様でした。今回、スコアは
  常に`evaluateProperty(p)`から計算する値に統一し、保存フィールドとしては
  廃止しました。物件データが変わるたびに再読み込みなしで最新のスコアが
  表示されます。

### Feature #007: 相場分析・価格帯学習

**目的**

これまでのScoreは「物件そのもの」の評価はできても、「価格が高いのか、
安いのか、妥当なのか」までは判断できませんでした。本Featureでは、
登録済みの物件データ同士を比較し、条件が近い物件の価格帯（最安・中央値
・最高）を可視化することで、物件を登録するほどProject:SS自身の相場
データが育っていく仕組みを目指しています。外部の相場データ・SUUMO等
との連携は使用していません。

**実装内容**

- **相場分析（`analyzeMarket(p)`）**：物件詳細画面に新設した「🏘 相場
  分析」カードから、その物件と条件が近い他の登録物件を自動的に探し、
  価格帯を計算します。比較条件は次の優先順位で、十分な件数（3件）が
  集まるまで自動的に広げていきます。
  1. エリア（住所の市区町村部分が一致）・築年数帯（10年刻み）・
     駐車場台数（対象物件以上）
  2. エリア・築年数帯
  3. エリアのみ
  4. 築年数帯のみ
  5. 登録済みの全物件（最終手段）
  
  条件を広げた場合は、モーダル内に「類似物件が少ないため条件を広げて
  います」と明示し、こっそり緩めた条件で表示することがないようにして
  います。今回のサンプル3件のように登録数が少ない段階では、多くの
  場合⑤（全物件）まで広がりますが、実物件を登録していくほど①〜④の
  より精度の高い条件で分析できるようになります＝「物件を登録するほど
  Project:SS自身が成長する」という仕様書の狙い通りの挙動です。
- **表示内容**：登録件数、最安値・中央値・最高値（万円）、価格帯コメント
  （「中央値より大幅に高めの価格帯です」など5段階）、分析対象になった
  物件の一覧（物件名・価格、リフォーム済みの場合はその旨のタグ）を、
  詳細画面の新しいカードとモーダルに表示しています。

   （現在の編集UIはそのまま活かしつつ）

**① 総合Score計算ロジックの見直し**

原因が分かりました。6つの評価軸（収益性・立地・建物・管理・法的リスク・
流動性・将来性）はそれぞれ内部的に0〜20点で計算していましたが、6軸
すべてが満点に近いと合計が100点を超えてしまうことがあり（最大
6×20＝120点）、その場合に総合Scoreだけを100点で強制的に頭打ちにして
いたため、「画面に表示されている内訳の合計」と「表示されている総合
Score」が一致しなくなっていました。

これを修正するため、各軸の内部計算（0〜20点、判定ロジック自体）は
そのまま変えず、最後に軸ごとの「配点」（収益性20点・立地15点・建物
管理20点・法的リスク20点・流動性15点・将来性10点＝合計ちょうど100点）
に換算し直す形にしました。各軸は必ず0〜配点の範囲に収まるため、
**総合Scoreは常に「画面に表示されている内訳の合計」と完全に一致する**
ことが計算の仕組みそのものによって保証されるようになりました（後から
別ロジックで丸める、という処理が無くなったため、原理的にズレが起き
得ません）。

レーダーチャート・内訳バー・AI詳細モーダルの表示も、軸ごとに異なる
配点（20点満点の軸と15点満点・10点満点の軸が混在）に対応させています。

**② 調査チェックをScoreへ反映**

実際には、雨漏り・シロアリ・再建築可否・接道状況・境界・災害リスクなど
主要な項目はすでにScoreの計算に使われていました。確認したところ、
2つの見落としがありました。

- 「建物の傾き」という調査項目自体が存在していませんでした。新しく
  調査チェックへ追加し（未確認／傾きなし／軽微な傾きあり／著しい傾き
  あり）、「建物・管理」の評価軸に組み込みました
- 「固定資産税」という調査項目は存在していましたが、どの評価軸からも
  参照されておらず、Scoreに一切反映されていませんでした。「流動性」の
  評価軸に組み込みました

「要確認」に該当する状態は、「問題あり」より軽い減点として扱う設計は
Feature #006の時点ですでに採用しており、今回追加した項目にもそのまま
適用されています。

**③ 調査チェック画面に閲覧モードを追加**

ご希望どおり、現在の調査チェックの編集UI（項目をタップ→選択肢シートが
開く）はまったく変更していません。その上で、「調査チェック」カードの
右上に「編集」ボタンを追加し、次のように動作するようにしました。

- 通常時（閲覧モード）：各項目はタップしても反応しない、読み取り専用の
  一覧として表示されます
- 「編集」をタップすると、これまでどおりの操作可能な一覧に切り替わり、
  ボタンの表示も「完了」に変わります
- 編集モード中に複数の項目を変更しても、その場ではScore・内訳・AI
  コメントは更新されません（調査完了率・未確認件数は、これまでどおり
  その場で更新されます）
- 「完了」をタップすると、Score・スコア内訳・AIコメントをまとめて
  再計算し、閲覧モードへ戻ります

ご依頼いただいた「基本情報→調査チェック→スコア内訳→総合Score→AI
コメント」という一連の流れが、この「完了」のタイミングで実際に連動する
形になっています。

**その他、あわせて見つけて直した点**

作業の過程で、直接は関係しない箇所の実際の不具合を1つ見つけました。
物件を新規に切り替えて詳細画面を開いたとき、前の物件で「編集」モードに
していた状態が残ってしまう可能性があったため、物件の詳細画面を開く
たびに必ず閲覧モードから始まるよう、明示的にリセットする処理を加えて
います。

### Feature #007 Follow-up Fix ⑤（AI詳細モーダルのスクロール位置リセット）

**ご依頼内容**

AIコメント詳細を開き、途中までスクロールした状態で閉じると、再度開いた
際に前回のスクロール位置が保持されてしまう。開くたびに先頭へリセット
してほしい、とのご依頼をいただきました。

**原因と修正内容**

モーダルを閉じたときに「元々見ていた詳細画面自体のスクロール位置」を
復元する処理はすでにありましたが、モーダル自身（AIコメントの中身）の
スクロール位置をリセットする処理が漏れていました。モーダルを開く処理に
1行追加し、開くたびに必ず先頭に戻るようにしました。

あわせて確認したところ、相場分析の詳細モーダルにも同じ抜けがあった
ため、同様に修正しています。

**セルフレビュー結果**

- [x] AI詳細モーダルを途中までスクロールしてから閉じ、再度開くと
      先頭に戻ることを実際にヘッドレスブラウザで確認
- [x] 相場分析モーダルにも同じ修正を適用し、同様に確認
- [x] 構文チェック（`node --check`）通過

- [ ] 実機iPhone Safariでの最終確認（未実施）
