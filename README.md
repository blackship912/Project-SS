# Project:SS — Beta 0.1

地方中古戸建て投資の物件評価・調査・学習ログを行う UI プロトタイプです。
「記録するアプリ」ではなく「使っているうちに記録されるアプリ」をコンセプトに、
物件のスコアリング・調査チェック・写真管理・比較・お気に入りなどを
HTML/CSS/JavaScript のみ（単一ファイル）で実装しています。

## Alpha と Beta の関係

**Project:SS Alpha 0.6.3 は Alpha シリーズの完成版として固定・保護されています。**
Alpha 0.6.3 のファイル自体は、これ以降編集しません。

このリポジトリ（Beta シリーズ）は、Alpha 0.6.3 をそのまま土台として、
新機能や UI 改善を積み重ねていくための開発ラインです。Alpha を壊さずに、
Alpha を基準として次の段階へ進むという考え方に基づいています。

- **Alpha 0.6.3**：完成版・セーブポイント。以後変更しない。
- **Beta 0.1〜**：ここから先の新機能・改善はすべてこちら側で行う。

Beta 0.1 は、Alpha 0.6.3 とタイトル・バージョン表記以外まったく同じ内容の
スタート地点です。まだ Beta 独自の変更は加えていません。

## GitHub Pages での公開方法

1. このリポジトリの内容（`index.html` と `.nojekyll`）をリポジトリのルートに配置します。
2. GitHub の **Settings → Pages** を開きます。
3. **Source** を `Deploy from a branch` に設定し、対象ブランチ（例: `main`）とフォルダ `/ (root)` を選択します。
4. 数分後、`https://<ユーザー名>.github.io/<リポジトリ名>/` でアクセスできるようになります。

`.nojekyll` は GitHub Pages のデフォルトの Jekyll 処理をスキップし、
ファイルをそのまま静的配信させるためのものです。

## Beta 0.1 での変更点

### Feature #001: 比較機能ブラッシュアップ

比較画面を「比較する画面」から「投資判断を支援する画面」に近づけるための改善です。
以下は既存のまま維持しています：左側の比較項目の固定表示、上部物件カードの通常表示
（横スクロールに追従）、物件写真・物件名タップでの詳細画面遷移、物件名の青文字＋下線
リンクデザイン、物件ごとのAI比較コメント、第1候補選択機能、ホーム画面の第1候補カード。

**① AI比較コメントUI改善**
- 一番スコアの高い物件のコメントだけ、最初から開いた状態で表示するようにしました。
  他の物件は今まで通りタップで開閉するアコーディオンのままです。これにより、何も
  操作しなくても少なくとも1件はすぐ読める状態になり、残りは必要に応じて確認できます。
- セクション見出しに「タップで開閉」という案内を添えました。

**② 比較項目の見直し**
- 投資判断への影響度が高い順に並び替えました：Score → 価格 → 利回り → 家賃 →
  調査完了率 → 未確認項目 → 築年数 → 構造 → 駐車場 → ステータス。
- 「家賃」（収益の根拠になる数値）と「未確認項目数」（リスクの残量、スコアの
  信頼度が分かる指標）を新しく追加しました。
- 「メモ数」「写真数」は投資判断そのものとは直接関係の薄い、アプリの利用状況に
  近い指標だったため、比較表からは外しました。
- 各比較項目を「ラベル・アイコン・値の取得方法・表示方法・優劣の判定方向」を
  1つにまとめた自己完結的な定義（`CMP_ROWS`）に整理し、今後の項目追加は配列に
  1つオブジェクトを足すだけで済むようにしました。AI比較コメントの強み・弱み分析も
  同じ定義を参照する仕組みに統合したので、新しい項目を追加すると自動的にAI
  コメントの分析対象にも含まれます（従来は比較表用とAIコメント用で項目定義が
  別々に存在していました）。

**③ 比較画面UX改善**
- 比較表に「📊 比較表」という見出しと、「横にスクロールできます」という案内を
  追加しました。
- 各比較項目のラベルに、内容がひと目で分かる小さいアイコンを添えました。
- セルの余白を広げ、文字サイズもわずかに大きくして読みやすくしました。
- 同点の項目には「優れている」ハイライトを付けないよう調整しました（以前は
  同点でも先頭の物件が強調されてしまう場合がありました）。

### Feature #002: Property Management Enhancement（物件削除機能）

対象 Issue：Issue #001 物件削除機能が未実装（Priority: High） / Status: Done

物件追加はできるのに削除ができなかった状態を解消し、実運用に耐えられるようにしました。

**実装内容**

- 物件詳細画面の一番下（ログタブの下、スワイプで切り替わる領域の外）に、
  「🗑️ この物件を削除」ボタンを追加しました。普段は目立たない控えめな
  グレーの文字リンクで、タップした瞬間だけ赤みが差します。危険な操作である
  ことは、ボタン自体ではなく次の確認ダイアログ側でしっかり伝える設計です。
- ボタンを押すと「この物件を削除しますか？」という確認シートが開きます。
  「写真・メモ・調査記録も削除されます。この操作は元に戻せません。」という
  補足と、「キャンセル」「削除」（赤字）の2つのボタンを用意しています。
  キャンセル・背景タップ・ハンドルの下スワイプでは何も削除されません。
- 「削除」を押すと、物件データそのもの（写真・写真のタグ・写真メモ・
  基本情報・調査チェック・メモ・ログ・お気に入り状態）をまとめて取り除きます。
  これらはすべて物件データ1つのオブジェクトの中にまとまっているため、
  そのオブジェクトごと配列から取り除くだけで、個別に消し忘れるということが
  起こらない作りになっています。
- 物件データ自体には含まれない、外部から物件IDを参照している状態
  （比較画面での選択状態、第1候補のID）は、削除のタイミングで明示的に
  クリアするようにしました。削除した物件が比較対象に入っていた場合は
  比較選択から外れ、第1候補に設定されていた場合は第1候補が解除され、
  ホーム画面の第1候補カードも表示されなくなります。
- 削除が完了すると、どの画面から詳細を開いていたかに関わらず、必ず
  ホーム画面に戻るようにしました。
- 物件が1件もない状態でもエラーが起きないようにしました。以前は物件が
  0件になると「注目物件」カードの表示処理でエラーが起きる作りになっていた
  ため、これも今回あわせて修正しています。物件が1件もない場合は、注目物件
  カード自体を非表示にし、物件一覧には「物件がまだありません／＋ボタンから
  物件を追加できます」という案内を表示するようにしました。

**セルフレビュー結果**

Python でのシミュレーションにより、お気に入り・比較対象・第1候補に入っていた
物件を削除した場合に、それぞれ正しく整理されることを確認しました。あわせて、
最後の1件を削除して0件になった状態から、新しい物件を1件追加するところまでの
一連の流れをシミュレーションし、IDの再利用によって古い状態が紛れ込まないことも
確認しています。構文チェック・ID整合性チェックも全通過しました。

- [x] 詳細画面から物件削除ボタンが使える
- [x] 削除確認ダイアログが表示される
- [x] キャンセルで削除されない
- [x] 削除を押すと物件が削除される
- [x] 削除後にホーム画面へ戻る
- [x] 削除済み物件が一覧に残らない
- [x] お気に入りからも消える
- [x] 比較対象からも消える
- [x] 第1候補だった場合、第1候補が解除される
- [x] ホーム画面の第1候補カードに削除済み物件が残らない
- [x] 最後の1件を削除してもエラーにならない
- [x] 既存の物件追加・編集・比較・写真機能を壊していない

**判断したこと**

- 削除操作のログ記録は見送りました。理由は、Project:SSのログは常に特定の
  物件に紐づく設計になっており、削除する物件自身にログを書き込んでも、
  その物件ごと消えてしまうため記録として意味を持たないためです。アプリ
  全体で共有するログのような仕組みは現状存在しないため、無理に新設する
  よりも、今回は見送るのが妥当と判断しました。
- 削除した物件のIDは、以後の新規登録で再利用され得る仕様のままにしています。
  削除時に比較選択・第1候補などの参照を確実にクリアしているため、IDが
  再利用されても古い物件の情報が紛れ込むことはありません。

**Improvement Ideas（追加改善提案）**

- 現状、削除は取り消せない一方向の操作です。将来的に「直前の削除を元に戻す」
  簡易的なアンドゥ機能があると、誤操作時の安心感が増すかもしれません。
- ※ 削除確認ダイアログのタイトルに削除対象の物件名を表示する改善は、
  提案にとどめず今回あわせて実装しました（「〇〇を削除しますか？」という
  形で、複数の物件を行き来していても削除対象を取り違えにくくしています）。

### Feature #003: Property Add Flow Cleanup（物件追加・基本情報編集フロー改善）

対象 Issue：
- Issue #003 物件追加フォームの入力内容が前回分のまま残る（Priority: High）
- Issue #004 入力画面のズーム復帰（Priority: High）
- Issue #005 物件追加・編集・基本情報表示の項目構成に差異がある（Priority: High）
- Issue #006 基本情報カードで未入力項目の単位だけが連続表示される（Priority: Medium）
- Issue #007 単位選択の2番目以降が選択できない（Priority: High）

**実装内容**

- **項目定義を1か所（`PROPERTY_FIELDS`）に統合しました。** これまで物件追加用・
  編集用・表示用でそれぞれ別々に項目のリストを持っており、しかも実際には
  項目構成が微妙に異なっていました（「所在地」「最寄駅」は表示画面にしかなく、
  追加・編集からは触れられない、といった状態）。今回、ラベル・アイコン・
  入力方式（数値／プルダウン／自由入力）・単位の選択肢を1つの項目につき
  1か所で定義する `PROPERTY_FIELDS` を新設し、新規登録フォーム・編集
  ウィザード・基本情報の表示グリッドの3つすべてがこの1つのリストから
  生成されるようにしました。物件名は詳細画面のヘッダーに既にあるため、
  このリストには含めていません。今後、項目を追加・変更・削除したい場合は
  `PROPERTY_FIELDS` を1か所直すだけで3画面すべてに反映されます。
  （なお、単位の選択肢一覧 `UNIT_OPTIONS` や、編集ウィザードのステップ分け
  `STEPS` も、すべてこの `PROPERTY_FIELDS` から自動的に導出される形に
  変更したため、二重管理の芽を作らないようにしています。）
- **新規登録フォームに前回の入力が残ってしまう問題**に対しては、以下の
  3段構えで対策しました。①「＋」を押すたびに、必ず空の状態のデータで
  フォームを作り直す、②フォームを閉じる（キャンセル／背景タップ／保存）
  いずれの経路でも、内部で保持しているフォームの下書きデータを確実に
  空へ戻す、③すべての入力欄に `autocomplete="off"` を指定し、ブラウザ
  自身が過去の入力を勝手に補完してしまう可能性を防ぐ。
- **入力後にズームしたまま戻らない問題**は、そもそもズームが起こらない
  ようにする方向で対策しました。iOS Safari は文字サイズが16pxより小さい
  入力欄をタップすると自動的にズームする仕様になっており、今回対象と
  なった入力欄の文字サイズが14pxだったため、これを16pxに引き上げました。
  ズームしてしまってから元に戻す、という後追いの対策ではなく、最初から
  ズームが発生しない状態を目指しています。
- **未入力項目の単位だけが連続表示される問題**は、数値項目の表示ロジックを
  「数値部分が実際に入力されている場合だけ、単位も含めた文字列を表示する」
  という判定に統一しました。あわせて、未入力の項目は今後すべて同じ「－」
  という記号1つで表示されるようにし、入力の有無がひと目でわかるように
  しています。
- **単位選択の2番目以降が選べない問題**については、単位の選択肢を変更した
  瞬間に、その場でフォーム全体を作り直す処理が同期的に走ることで、iOS
  Safari側の内部的な選択状態の確定処理と衝突している可能性を想定し、
  この作り直し処理を選択操作の直後ではなく、わずかに遅らせて実行する
  ように変更しました。これはブラウザでこの種の不具合を避ける際によく
  使われる対策です。ただし、この環境では実機のiOS Safariで直接動作を
  確認する手段がないため、確実に解消したと言い切ることはできません。
  改善後も同じ症状が起きる場合は、その旨をお知らせいただけると、別の
  角度から原因を調査します。

**セルフレビュー結果**

Node.js 上でフィールド定義から実際に生成される新規登録フォーム・編集
ウィザード・表示グリッドの項目キーを突き合わせ、物件名とステータスを除いて
完全に一致することを確認しました。既存の3件のサンプル物件データと、
真っ新な新規物件（すべての項目が空）の両方について、表示ロジックが
それぞれ正しい文字列と「－」を返すことも確認しています。

- [x] 新規物件追加時に前回入力内容が残らない（フォームの作り直し・明示的な
      クリア・autocomplete対策の3段構え）
- [x] 新規物件追加・基本情報編集ともに入力欄の文字サイズを16px化（ズームの
      発生自体を防止）
- [x] 物件追加・編集・基本情報表示（物件名を除く）の項目が単一の定義から
      生成され、完全に一致することを確認
- [x] 項目定義が保守しやすい構造になっている（`PROPERTY_FIELDS` 1か所に統合）
- [x] 未入力項目は「－」で統一表示される
- [x] 単位だけが連続表示されない（数値が空なら値自体を表示しない判定に変更）
- [ ] すべての単位が正常に選択できる（一般的な回避策は実装したが、実機での
      最終確認はできていない）
- [x] 保存内容が正しく基本情報へ反映される
- [x] 既存機能への影響を確認（既存3件のサンプルデータの表示・保存ロジックを
      個別に検証）

**Improvement Ideas（追加改善提案）**

- 「所在地」「最寄駅・徒歩」は今回はじめて新規登録・編集フォームに組み込んだ
  項目です。過去に登録された物件データはこれらの項目を既に持っていますが、
  改めて実際の編集画面で正しく開いて保存できるか、実機での確認をお願い
  したいです。
- 検索窓（ホーム画面）や写真タグの自由入力欄も、今回の対象画面と同じ
  「14px未満のためズームしてしまう」条件に当てはまっています。今回は
  Issueで指定された2画面に絞って対応しましたが、同じ理由でズームが
  気になる場合は、これらの入力欄も合わせて16px化するとよいと思います。

### Feature #003 Follow-up Fix（実機確認による追加修正）

実機での確認により、前回の対応だけでは不十分だった点と、対応範囲が
不足していた点が見つかったための追加修正です。

対象 Issue：
- Issue #003（Reopen）物件追加フォームの入力内容が前回分のまま残る
- Issue #004（Scope Expanded）入力画面のズーム復帰・ズーム抑制対応
- Issue #006（Reopen / Scope Expanded）単位だけが連続表示される

**修正内容**

- **Issue #003（フォーム残留・再発）**：前回はフォームを開くたびに新しい
  空のデータで作り直す処理と、`autocomplete="off"`の指定で対応していま
  したが、それでも残る場合があるとのことでした。iOS Safari には、
  同じ`id`を持つ入力欄がページ内で作り直されても、ブラウザ自身が過去の
  入力を独自に復元してしまうことがある、という既知の癖があります。
  今回、マークアップ側の初期値だけに頼るのではなく、フォームを開いた
  直後に、すべての入力欄・プルダウン・単位選択の`.value`（実際に画面に
  表示されている値）をJavaScript側から強制的に空へ書き換える処理を
  追加しました。基本情報の編集画面についても、開いた際に既存のデータで
  同様に値を明示的に同期させる処理を加えています。これにより、たとえ
  ブラウザが独自に値を復元しようとしても、その直後に確実に正しい状態へ
  上書きされます。
- **Issue #006（単位連続表示・適用範囲拡大）**：基本情報カード用に作った
  「数値が実際にある場合だけ単位も表示する」という判定ロジックを、
  キー名を指定するだけでどこからでも呼び出せる共通の仕組み
  （`propFieldValue` / `propFieldValueOrDash`）として独立させました。
  これを使って、写真の下に出る価格・利回り・築年数の表示、比較画面の
  価格・利回り・家賃・築年数・構造・駐車場の表示、ホーム画面の物件
  一覧カード・お気に入り一覧カード・第1候補カードの価格・利回り表示、
  合わせて8か所を同じ表示ルールに揃えました。これらはいずれも今回
  あらためて確認するまで見落としていた箇所で、基本情報カードだけを
  直しても同じ問題が他の画面に残っている状態でした。
- **Issue #004（ズーム対策・全入力欄への適用）**：これまで新規物件追加と
  基本情報編集の入力欄だけをズーム対策済みの16pxにしていましたが、
  今回、検索欄・メモ入力・メモ編集フォーム・見送り理由入力・写真タグの
  自由入力・写真メモ入力も含め、アプリ内のすべての入力欄を16px以上に
  揃えました。これにより、ズームしてから元に戻すという後追いの対応
  ではなく、そもそもどの入力欄をタップしてもズームが起こらない状態に
  なっています。

**セルフレビュー結果**

- [x] 新規物件追加フォームを何度開いても前回入力内容が残らない（開く
      たびの再生成に加えて、DOM上の値を明示的に上書きする処理を追加）
- [x] 数値・テキスト・プルダウン・手入力欄・単位選択がすべて初期化される
- [x] 表示値（DOM上の`.value`）と内部状態（下書きデータ）の両方を
      初期化する処理にした
- [x] 基本情報カードで単位だけが連続表示されない
- [x] 写真下の3項目で単位だけが連続表示されない
- [x] 比較画面で単位だけが連続表示されない
- [x] 値がない項目は「－」で統一表示される（8か所すべてで共通の関数を使用）
- [x] 入力欄の文字サイズを全箇所16px以上に統一し、入力時ズームが

**重要な注意事項（実機確認前に必ず確認いただきたいこと）**

写真はこれまで通り base64 文字列としてメモリ上に保持されており、今回の
自動保存でもその形のまま localStorage に書き込まれます。iPhone Safari の
localStorage には端末ごとに容量上限があり、一般的には数MB程度とされて
います。実物件の写真（特にスマートフォンで撮影したそのままの高解像度
画像）を数枚保存しただけで、この上限に達する可能性があります。

上限に達した場合でもアプリがクラッシュしたり物件データそのものが壊れたり
することはなく、自動保存だけが失敗し、設定画面の自動保存ステータスが
「⚠️ 自動保存に失敗しました」という表示に切り替わる設計にしています
（メモリ上のデータ自体は保持されるため、その場でのアプリ利用は継続でき
ます）。ただし、その状態でアプリを閉じたりリロードしたりすると、直近の
変更内容が保存されずに失われる可能性があります。

実機確認の際は、ぜひ実際の物件写真を数枚登録した状態で自動保存ステータスの
表示を確認していただきたいです。もし警告表示が出るようであれば、次の
Featureで写真を保存前に縮小・圧縮する対応を検討する必要があります
（Improvement Ideas にも記載します）。

**Improvement Ideas（追加改善提案）**

- 写真を保存前にリサイズ・圧縮してからメモリ・localStorageに保持する
  対応。現状は撮影した写真をそのままの解像度で保持しているため、実機で
  複数枚の写真を扱う運用が始まると、上記の容量上限に到達しやすくなる
  懸念があります。
- 自動保存に失敗した状態がしばらく続いた場合に、設定画面を開かなくても
  気づけるような通知（詳細画面やホーム画面での軽い警告表示など）が
  あると、より安心感が増すかもしれません。
- バックアップファイルに任意のメモ（「購入前の最終バックアップ」など）
  を付けられると、複数のバックアップを取った際に見分けやすくなります
  （現状は複数バックアップの管理自体が対象外のため、次のステップとして）。

   意味のある変更のほぼ全箇所で呼ばれている）に保存処理を統合しており、
   新規登録・削除・元に戻す・データ初期化・バックアップ復元の5箇所は
   個別に保存呼び出しを追加済みであることをコード上で再確認しました。
3. 初期化時に`loadPersistedProps()`の結果がサンプルデータより優先
   されているか → 優先されています。保存データがあれば必ずそちらを
   使う実装になっています。
4. localStorageの`STORAGE_KEY`が保存時・読込時で一致しているか →
   一致しています（`"project_ss_props_v1"`を1箇所で定義し、保存・読込
   両方から参照）。
5. iPhone Safari環境でlocalStorage保存が失敗していないか → **これは
   Claude側では直接確認できません**（4.3節の通り、実機での動作確認は
   竜真さんの役割です）。ただし、今回の症状には重要な手がかりがあります。
   「バックアップ作成」「バックアップ復元」はどちらも localStorage を
   一切使っていません（ファイルのダウンロード・読み込みのみで完結する
   仕組みです）。この2つが正常動作したという事実は、「localStorageが
   確実に動いている」ことの証明にはならず、むしろ**localStorage
   自体が使えない環境である可能性**を消せていません。

**見つけて修正した実際の不具合**

5項目のご確認とは別に、コードを洗い直す過程で、実際の不具合を1つ
見つけました。

`log()`関数は物件データの変更箇所のほとんどから呼ばれていますが、その
呼ばれるタイミングは箇所によってまちまちで、「実際のデータ変更より先に
`log()`が呼ばれている」箇所が2つありました（写真削除、基本情報編集
ウィザードでの複数項目保存）。これらの箇所で`log()`の中から直接
`persistProps()`を呼んでいたため、保存される内容が「その変更が反映
される直前」の、1つ古い状態になってしまう不具合でした。

これに対し、`log()`が呼ぶ保存処理を「次のtick」へ遅延させる
`schedulePersist()`という仕組みに変更しました。これにより、`log()`が
データ変更の前に呼ばれるか後に呼ばれるかに関わらず、実際に保存される
のは必ずその場の一連の処理がすべて完了した後の状態になります。
Node.js上でこの挙動をシミュレーションし、修正後は写真削除操作の直後の
保存内容に、削除後の状態（削除された写真を含まない）が正しく反映される
ことを確認しました。

ただし、この不具合は「特定の1項目・1枚の写真だけが1つ前の状態に戻る」
という症状であり、竜真さんが実機で見た「リロードで全体が初期状態に
戻る」という症状とは一致しません。そのため、この修正だけでは今回の
症状は説明しきれないと考えています。

**もう一つの対応：保存失敗を必ず目に見える形にする**

上記5番の通り、Claude側では実機のlocalStorageの動作を直接確認できない
ため、原因調査に協力いただく前提で、まず「保存に失敗していたら絶対に
見逃さない」ようにする対応を先に入れました。これまでは保存失敗時、
設定画面の「データ管理」を開かない限り気づけない小さな表示だけでした
が、今回、保存に失敗している間はどの画面を開いていても常に見える赤い
警告バナーを画面上部に追加しました。

**竜真さんに確認いただきたいこと**

お手数ですが、次の3点を確認いただけますでしょうか。

1. 今回追加した画面上部の赤い警告バナー（「⚠️ 自動保存ができていません」）
   が表示されているかどうか。表示されていれば、原因はコードの不具合
   ではなく、iPhone Safari側でこの端末の保存機能が使えない状態に
   なっている可能性が高いです。
2. GitHub Pages の URL（`https://...github.io/...`）経由で開いています
   か、それともファイルを直接（AirDrop・Files アプリなどから）開いて
   いますか。
3. Safari のプライベートブラウズタブで開いていないかどうか。また、
   「設定 > Safari > すべての Cookie をブロック」がオンになっていない
   かどうか（この設定はオンにすると Cookie だけでなく localStorage も
   まとめてブロックされます）。
